## ⚙️ 개발 환경 오류 체크리스트 (Debugging Guide)

### 🔴 Level 1: 환경 및 의존성 누락 오류 (Missing Dependencies/Environment)
이 단계의 오류는 **코드가 실행되기도 전에** 발생하는 문제입니다. "이게 아예 돌지도 않네?" 싶을 때가 많습니다.

| 원인 (Why it breaks) | 상세 설명 및 발생 이유 | 해결책 (How to fix) | 체크 포인트 (Verification) |
| :--- | :--- | :--- | :--- |
| **1. API 키/시크릿 누락** | 외부 서비스(OpenAI, AWS 등)를 호출할 때, 환경 변수(`process.env.API_KEY`)에 키가 설정되지 않았거나, 키 자체가 만료/오타인 경우. | 1. **`.env` 파일 사용:** 프로젝트 루트에 `.env` 파일을 만들고 `KEY=value` 형태로 저장합니다. 2. **`.gitignore`에 등록:** `.env` 파일이 Git에 올라가지 않도록 반드시 등록해야 합니다. 3. **코드에서 로드:** `dotenv` 등의 라이브러리를 사용해 환경 변수를 로드하도록 코드를 수정합니다. | ✅ `process.env.YOUR_KEY`를 `console.log`로 찍어보고 `undefined`가 아닌지 확인하세요. |
| **2. 패키지 미설치** | 필요한 라이브러리(예: `requests`, `pandas`, `next`)가 현재 실행 환경에 설치되어 있지 않은 경우. | 1. **Python:** `pip install -r requirements.txt` (가장 확실한 방법) 2. **Node.js:** `npm install` 또는 `yarn install` 3. **특정 패키지:** `npm install [package-name]` | ✅ **`node_modules` 또는 `venv` 폴더 확인:** 해당 패키지가 실제로 폴더에 존재하는지, 그리고 `package.json`에 명시되어 있는지 확인합니다. |
| **3. 종속성 버전 충돌** | A 패키지는 버전 1.0을 요구하는데, B 패키지가 2.0을 요구하여 호환되지 않는 경우. (Dependency Hell) | 1. **버전 명시:** `package.json` 또는 `requirements.txt`에서 명시적인 버전을 지정합니다. (예: `react: ^18.2.0`) 2. **호환성 점검:** 문제가 되는 패키지의 GitHub 이슈나 문서를 찾아 호환성 문제를 먼저 파악합니다. | ✅ `npm outdated` 또는 `pip check` 명령어로 패키지 간의 알려진 충돌을 검색해 보세요. |

### 🟡 Level 2: 설정 및 타입 불일치 오류 (Configuration/Type Mismatch)
코드가 돌기는 하지만, "이상한 값이 나온다"거나 "예상치 못한 에러"가 발생하는 단계입니다. 가장 흔하게 개발자가 간과하는 부분입니다.

| 원인 (Why it breaks) | 상세 설명 및 발생 이유 | 해결책 (How to fix) | 체크 포인트 (Verification) |
| :--- | :--- | :--- | :--- |
| **1. 잘못된 데이터 타입 사용** | API 응답이 숫자인 줄 알고 사용했는데, 실제로는 문자열(`"123"`)인 경우. Python에서 `int`를 기대했는데 `str`이 들어오면 계산이 안 됩니다. | 1. **명시적 형 변환:** 데이터를 사용할 때 반드시 `int()`, `str()`, `parseFloat()` 등을 이용해 타입을 명시적으로 변환합니다. 2. **타입 스키마 정의:** TypeScript나 Pydantic 같은 런타임 타입 검사 라이브러리를 사용해 데이터 구조를 강제합니다. | ✅ 해당 변수가 어떤 타입(`typeof variable` 또는 `type`)을 가질지 예측하고, 가장 안전한 기본값(Default Value)을 설정하세요. |
| **2. 비동기 처리 누락** | `await` 키워드를 빼먹거나, `async` 함수 내부에서 비동기 함수를 호출하고 결과를 기다리지 않는 경우. (Promise가 제대로 해결되지 않음) | 1. **`await` 사용:** I/O 작업(DB 접근, API 호출 등)을 할 때는 반드시 `await`을 사용하여 작업 완료를 기다려야 합니다. 2. **`try...catch` 블록:** 비동기 작업은 실패할 가능성이 높으므로, 반드시 `try...catch`로 감싸서 에러를 잡을 준비를 해야 합니다. | ✅ 비동기 함수를 호출하는 라인에 `await`이 붙어 있는지, 그리고 그 코드가 `async` 함수 안에 있는지 확인하세요. |
| **3. 경로(Path) 문제** | 파일 시스템 경로나 URL 경로에 오타가 있거나, 운영체제별 경로 구분자(`\` vs `/`) 차이를 인지하지 못한 경우. | 1. **OS 독립적 경로 사용:** 파이썬의 `os.path.join()`이나 Node.js의 `path.join()` 같은 OS-agnostic 라이브러리를 사용해 경로를 조합하세요. 2. **절대 경로 사용:** 로컬 개발 환경에서는 절대 경로를 사용하고, 배포 환경에서는 상대 경로를 사용하도록 로직을 분리합니다. | ✅ 경로 변수를 `console.log`로 출력해보고, 예상했던 OS 구조와 일치하는지 시각적으로 확인하세요. |

### 🚨 Level 3: 배포 및 런타임 환경 오류 (Deployment/Runtime)
로컬(Local)에서는 완벽했는데, 서버(Server)에 올리니 깨지는 경우입니다. 이 오류들은 **환경 차이**가 핵심입니다.

| 원인 (Why it breaks) | 상세 설명 및 발생 이유 | 해결책 (How to fix) | 체크 포인트 (Verification) |
| :--- | :--- | :--- | :--- |
| **1. 환경 변수의 불일치** | 로컬 `.env` 파일에는 키가 있는데, 배포 서버의 환경 변수 설정에 누락된 경우. | 1. **환경 변수 명세화:** 모든 필수 환경 변수 목록을 문서화하고, CI/CD 파이프라인에서 이 목록을 검증하도록 합니다. 2. **기본값 설정:** 코드가 환경 변수가 없을 경우를 대비하여 기본값(Fallback Value)을 가지도록 코드를 작성합니다. | ✅ 배포 서버의 환경 변수 설정 화면을 열어, 로컬과 동일한 모든 키가 정의되어 있는지 눈으로 확인하세요. |
| **2. 권한 문제 (Permissions)** | 서버 프로세스가 특정 디렉토리(`uploads/`, `data/`)에 파일을 쓰거나 읽을 권한이 없는 경우. | 1. **권한 확인:** 서버 OS에서 해당 디렉토리에 대한 쓰기/읽기 권한(chmod/chown)을 `www-data` 또는 실행 사용자 계정에 부여합니다. 2. **서비스 계정 분리:** 웹 애플리케이션과 데이터베이스 접근을 위한 계정을 분리하고, 최소 권한 원칙(Principle of Least Privilege)을 적용합니다. | ✅ 서버에서 해당 폴더에 임시 파일을 생성해보거나, `ls -l`로 권한을 확인해 보세요. |
| **3. 메모리/리소스 제한** | 대용량 데이터를 처리할 때 서버의 메모리(RAM)나 CPU 자원을 초과하는 경우. | 1. **청크(Chunk) 처리:** 대용량 파일이나 데이터셋을 한 번에 로드하지 않고, 작은 덩어리(Chunk)로 나누어 처리합니다. 2. **최적화:** 불필요한 데이터 전송이나 메모리 할당을 줄이는 알고리즘 최적화를 수행합니다. | ✅ 서버의 모니터링 도구(예: `top`, AWS CloudWatch)를 켜놓고, 피크 타임에 리소스 사용량을 측정하며 테스트합니다. |

| 원인 (Why it breaks) | 상세 설명 및 발생 이유 | 해결책 (How to fix) | 체크 포인트 (Verification) |
| :--- | :--- | :--- | :--- |
| **1. 잘못된 데이터 타입 사용** | API 응답이 숫자인 줄 알고 사용했는데, 실제로는 문자열(`"123"`)인 경우. Python에서 `int`를 기대했는데 `str`이 들어오면 계산이 안 됩니다. | 1. **명시적 형 변환:** 데이터를 사용할 때 반드시 `int()`, `str()`, `parseFloat()` 등을 이용해 타입을 명시적으로 변환합니다. 2. **타입 스키마 정의:** TypeScript나 Pydantic 같은 런타임 타입 검사 라이브러리를 사용해 데이터 구조를 강제합니다. | ✅ 해당 변수가 어떤 타입(`typeof variable` 또는 `type`)을 가질지 예측하고, 가장 안전한 기본값(Default Value)을 설정하세요. |
| **2. 비동기 처리 누락** | `await` 키워드를 빼먹거나, `async` 함수 내부에서 비동기 함수를 호출하고 결과를 기다리지 않는 경우. (Promise가 제대로 해결되지 않음) | 1. **`await` 사용:** I/O 작업(DB 접근, API 호출 등)을 할 때는 반드시 `await`을 사용하여 작업 완료를 기다려야 합니다. 2. **`try...catch` 블록:** 비동기 작업은 실패할 가능성이 높으므로, 반드시 `try...catch`로 감싸서 에러를 잡을 준비를 해야 합니다. | ✅ 비동기 함수를 호출하는 라인에 `await`이 붙어 있는지, 그리고 그 코드가 `async` 함수 안에 있는지 확인하세요. |
| **3. 경로(Path) 문제** | 파일 시스템 경로나 URL 경로에 오타가 있거나, 운영체제별 경로 구분자(`\` vs `/`) 차이를 인지하지 못한 경우. | 1. **OS 독립적 경로 사용:** 파이썬의 `os.path.join()`이나 Node.js의 `path.join()` 같은 OS-agnostic 라이브러리를 사용해 경로를 조합하세요. 2. **절대 경로 사용:** 로컬 개발 환경에서는 절대 경로를 사용하고, 배포 환경에서는 상대 경로를 사용하도록 로직을 분리합니다. | ✅ 경로 변수를 `console.log`로 출력해보고, 예상했던 OS 구조와 일치하는지 시각적으로 확인하세요. |

### 🚨 Level 3: 배포 및 런타임 환경 오류 (Deployment/Runtime)
로컬(Local)에서는 완벽했는데, 서버(Server)에 올리니 깨지는 경우입니다. 이 오류들은 **환경 차이**가 핵심입니다.

| 원인 (Why it breaks) | 상세 설명 및 발생 이유 | 해결책 (How to fix) | 체크 포인트 (Verification) |
| :--- | :--- | :--- | :--- |
| **1. 환경 변수의 불일치** | 로컬 `.env` 파일에는 키가 있는데, 배포 서버의 환경 변수 설정에 누락된 경우. | 1. **환경 변수 명세화:** 모든 필수 환경 변수 목록을 문서화하고, CI/CD 파이프라인에서 이 목록을 검증하도록 합니다. 2. **기본값 설정:** 코드가 환경 변수가 없을 경우를 대비하여 기본값(Fallback Value)을 가지도록 코드를 작성합니다. | ✅ 배포 서버의 환경 변수 설정 화면을 열어, 로컬과 동일한 모든 키가 정의되어 있는지 눈으로 확인하세요. |
| **2. 권한 문제 (Permissions)** | 서버 프로세스가 특정 디렉토리(`uploads/`, `data/`)에 파일을 쓰거나 읽을 권한이 없는 경우. | 1. **권한 확인:** 서버 OS에서 해당 디렉토리에 대한 쓰기/읽기 권한(chmod/chown)을 `www-data` 또는 실행 사용자 계정에 부여합니다. 2. **서비스 계정 분리:** 웹 애플리케이션과 데이터베이스 접근을 위한 계정을 분리하고, 최소 권한 원칙(Principle of Least Privilege)을 적용합니다. | ✅ 서버에서 해당 폴더에 임시 파일을 생성해보거나, `ls -l`로 권한을 확인해 보세요. |
| **3. 메모리/리소스 제한** | 대용량 데이터를 처리할 때 서버의 메모리(RAM)나 CPU 자원을 초과하는 경우. | 1. **청크(Chunk) 처리:** 대용량 파일이나 데이터셋을 한 번에 로드하지 않고, 작은 덩어리(Chunk)로 나누어 처리합니다. 2. **최적화:** 불필요한 데이터 전송이나 메모리 할당을 줄이는 알고리즘 최적화를 수행합니다. | ✅ 서버의 모니터링 도구(예: `top`, AWS CloudWatch)를 켜놓고, 피크 타임에 리소스 사용량을 측정하며 테스트합니다. |
